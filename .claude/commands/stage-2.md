## 2단계: 사업성/기술/경쟁력 평가

대상 아이디어: $ARGUMENTS

## 목적
6개 항목(MVP 출시 가능성, Pain Point 명확성, 기술 스택 적합성, 시장 규모, 경쟁 강도, 수익 모델 명확성)을 정량 평가한다.

## 입력
- `ideas/`에서 대상 아이디어 문서를 읽는다
- `research/market-data/NNN-*.md` 파일이 있으면 참고한다 (NNN은 대상 아이디어 번호)
- `research/competitors/NNN-*.md` 파일이 있으면 참고한다

## 사용 도구
| 도구 | 역할 |
|------|------|
| MCP: `naver-search` | 네이버 블로그/카페/쇼핑 데이터로 한국 시장 수요·불만 검증 |
| MCP: `brave-search` | 시장 규모, 경쟁사 정보, 앱 스토어 데이터, 뉴스 수집 |
| MCP: `fetch` | 경쟁 앱 스토어 페이지, 랜딩페이지, 가격 정책 페이지 직접 확인 |

## 절차

### 1. 데이터 수집
- `naver-search`로 네이버 블로그·카페·커뮤니티에서 관련 키워드의 실제 불만/수요 게시글 수집
- `brave-search`로 시장 규모 통계, 경쟁 앱 정보, 업계 뉴스 수집
- `fetch`로 경쟁 앱 스토어 페이지·랜딩페이지를 직접 읽어 기능 목록, 가격표, 사용자 리뷰 확인

### 2. 분석 (수집 데이터 기반으로 직접 수행)

**시장 규모**: TAM → SAM → SOM 단계적 추정. 각 단계마다 산출 논거를 명시한다.

**경쟁 환경**: 직접 경쟁자 3개 이내 식별. 각각의 강점·약점·차별점을 정리한다. 추가로 진입장벽(대기업·기존 앱의 네트워크 효과, 전환 비용, 데이터 해자)과 대체재 위협을 평가한다.

**고객 직무(JTBD)**: 고객이 해결하려는 핵심 직무를 정의하고, 기존 대안이 해당 직무를 과잉 충족 또는 부족 충족하는 지점을 식별한다. 네이버 블로그/카페/커뮤니티에서 수집한 실제 불만·수요 근거를 인용한다.

**수익화**: 유사 앱의 과금 모델을 참고하여 사용자 지불 의향을 판단한다.

### 3. 평가
- 아래 점수 기준표에 따라 6개 항목 점수 기입 (각 항목 1-5점, 총 30점 만점)
- 총점에 따른 판정 결과를 사용자에게 보고

#### 점수 기준표

**1. MVP 출시 가능성 (1-3개월)**
| 점수 | 기준 |
|------|------|
| 5 | 1개월 내 핵심 기능 구현 가능 |
| 4 | 2개월 내 구현 가능 |
| 3 | 3개월 내 구현 가능 |
| 2 | 3개월 초과 예상 |
| 1 | 1인 개발로 MVP 자체가 어려움 |

**2. Pain Point 명확성**
| 점수 | 기준 |
|------|------|
| 5 | 커뮤니티에서 반복적으로 언급되는 불만 |
| 4 | 특정 그룹에서 명확한 수요 확인 |
| 3 | 논리적으로 수요가 추론됨 |
| 2 | 있으면 좋지만 급하지 않은 문제 |
| 1 | 만든 사람만 쓸 것 같음 |

**3. 기술 스택 적합성**
| 점수 | 기준 |
|------|------|
| 5 | RN + Supabase만으로 완전 구현 |
| 4 | 소수 외부 API 연동 필요 (지도 등) |
| 3 | 결제/인증 등 외부 서비스 필요 |
| 2 | 네이티브 모듈 또는 복잡한 외부 연동 필요 |
| 1 | RN으로 구현 부적합 (AR, 고성능 그래픽 등) |

**4. 시장 규모**
| 점수 | 기준 |
|------|------|
| 5 | 한국 전체 성인 대상 (수천만) |
| 4 | 대형 세그먼트 (직장인, 대학생 등) |
| 3 | 중형 세그먼트 (특정 직군, 취미 등) |
| 2 | 소형 니치 (수만 명 수준) |
| 1 | 극소수 대상 |

**5. 경쟁 강도**
| 점수 | 기준 |
|------|------|
| 5 | 직접 경쟁자 없음 |
| 4 | 경쟁자 있으나 명확한 차별점 가능 |
| 3 | 경쟁자 있고 차별점이 애매함 |
| 2 | 대기업(네이버/카카오) 유사 서비스 존재 |
| 1 | 네이버/카카오가 핵심 기능으로 제공 중 |

**6. 수익 모델 명확성**
| 점수 | 기준 |
|------|------|
| 5 | 사용자가 자연스럽게 결제할 구조 |
| 4 | 프리미엄/구독 모델 적용 가능 |
| 3 | 광고 기반 수익 가능 |
| 2 | 수익화 경로가 불분명 |
| 1 | 수익화 방법을 모르겠음 |

#### 판정 기준
| 총점 | 판정 |
|------|------|
| 24-30 | **채택** - 즉시 기획 착수 |
| 18-23 | **보류** - 추가 조사 후 재평가 |
| 12-17 | **탈락** - 사유 기록 후 아카이브 |
| ~11 | **탈락** - 근본적 제약 있음 |

## 산출물
| 파일 | 위치 |
|------|------|
| 평가 완료 아이디어 문서 | `ideas/NNN-아이디어명.md` (점수 기입 완료) |
| 경쟁사 분석 | `research/competitors/NNN-경쟁분석.md` |
| 시장 분석 | `research/market-data/NNN-시장분석.md` |

## 전환
- 24-30점 채택 → "3단계: `/stage-3 NNN-아이디어명`" 안내
- 18-23점 보류 → 미달 항목 보고 후 사용자 판단 요청 (추가 조사 / 피벗 / 탈락 / 예외 채택)
- ~17점 탈락 → 탈락 처리 수행 (사용자 동의 필수)

### 보류 흐름 (18-23점)
1. 미달 항목을 식별하여 사용자에게 보고
2. 사용자 판단에 따라:
   - **추가 조사**: 미달 항목을 보강한 뒤 2단계 재평가
   - **피벗**: 아이디어 방향을 수정하여 1단계로 회귀
   - **탈락 처리**: 사용자 동의 후 탈락 처리 수행
   - **예외 채택**: 사용자가 명시적으로 점수와 무관하게 3단계 진행 결정

### 탈락 처리
1. **반드시 사용자의 명시적 동의** 확인
2. 아이디어 문서 판정란에 `탈락` 기록 — 탈락 단계·미달 항목·사유를 판정란에 직접 작성 (별도 rejection 파일 생성하지 않음)
3. 관련 파일(`NNN-아이디어명.md`, 기존 리서치 파일 등)을 `ideas/rejected/`로 **이동** (원본 삭제)

### 보류 처리
- `ideas/` 루트에 유지, 판정란에 `보류` 기록 및 미달 항목·추가 조사 방향 명시
- 재개 시 `/stage-2 NNN-아이디어명`부터 재시작
