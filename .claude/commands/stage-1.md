## 1단계: 아이디어 발굴

$ARGUMENTS

## 목적
한국 시장의 실제 pain point를 기반으로 앱 아이디어를 도출한다.

## 사용 도구
| 도구 | 역할 |
|------|------|
| MCP: `naver-search` → `search_blog` | 네이버 블로그에서 사용자 불만·후기 수집 |
| MCP: `naver-search` → `search_cafearticle` | 네이버 카페에서 커뮤니티 토론·수요 수집 |
| MCP: `naver-search` → `search_news` | 최신 뉴스·트렌드 키워드 수집 |
| MCP: `naver-search` → `search_kin` | 지식iN에서 구체적 질문·고충 수집 |
| MCP: `naver-search` → `datalab_search` | 키워드 검색 추세 검증 |
| MCP: `brave-search` → `brave_web_search` | 글로벌 트렌드 및 해외 유사 앱 탐색 |
| MCP: `fetch` | 검색 결과 중 유의미한 URL의 본문 상세 확인 |

## 절차

### 1. 주제 결정

**$ARGUMENTS가 있는 경우:**
입력된 주제를 그대로 탐색 주제로 사용하여 절차 2번으로 진행한다.

**$ARGUMENTS가 없는 경우 — 트렌드 기반 자동 분석:**
1. `search_news`로 최근 1주일 화제 키워드·뉴스 수집 (앱으로 해결 가능한 생활 밀착 주제 중심)
2. 1에서 발견한 유망 키워드를 `datalab_search`에 투입하여 검색량 추세 상승 여부 검증
3. `brave_web_search`로 글로벌 앱 트렌드 및 Product Hunt 신규 앱 확인 (1~2와 병렬 실행)
4. 수집된 트렌드를 분석하여 앱 아이디어로 발전 가능한 후보 3개를 도출
5. 후보 3개를 사용자에게 텍스트로 제시하고, AskUserQuestion으로 선택받아 절차 2번 진행

### 2. 검색 쿼리 구성

주제가 결정되면 아래 패턴으로 검색 쿼리를 구성한다:
- **불만·수요 탐색**: `"{주제} 불편"`, `"{주제} 힘들다"`, `"{주제} 추천 앱"`, `"{주제} 방법 없나"`
- **기존 솔루션 탐색**: `"{주제} 앱 추천"`, `"{주제} 앱 비교"`, `"{주제} 앱 단점"`
- **글로벌 유사 앱**: `"{주제 영문} app"`, `"{주제 영문} app alternative"`

### 3. Pain point 수집 (병렬 실행)

아래 두 그룹은 **병렬로 동시 실행**한다:

**그룹 A — 한국 사용자 불만·수요 (naver-search):**
- `search_blog`: 불만·후기 키워드로 실제 사용자 경험 수집
- `search_cafearticle`: 커뮤니티 토론에서 반복되는 pain point 파악
- `search_kin`: "~하려면 어떻게 해야 하나요" 류의 구체적 고충 수집

**그룹 B — 글로벌 트렌드·경쟁 (brave-search):**
- `brave_web_search`: 해외 유사 앱·서비스 존재 여부 및 트렌드 확인

### 4. 근거 심화 (fetch)

절차 3에서 수집한 검색 결과 중 pain point 근거로 유의미한 블로그·기사·게시글 URL은 `fetch`로 본문을 확인하여 구체적 인용 근거를 확보한다.

### 5. 사전 필터링

제외 기준에 해당하는지 점검한다:
- 법적 인허가 필수 (의료, 금융 등)
- 기업/기관 협약 필수
- 1인 개발 불가

기술 스택 적합성을 점검한다:
- Expo + RN + Supabase로 핵심 기능 구현 가능한가
- Windows 11 + EAS Build 환경에서 개발·테스트·빌드가 가능한가 (macOS 필수 도구·iOS 전용 API 과의존 여부)
- 플랫폼 전용 기능(HealthKit, ARKit 등)에 핵심이 의존하지 않는가

→ 제외 기준 해당 또는 기술 스택 부적합 시 폐기 후 새 아이디어 발굴. 모두 통과 시 절차 6번 진행.

### 6. 초안 작성

아래 초안 작성 지침에 따라 아이디어 초안을 작성한다.

## 초안 작성 지침

### NNN 번호 부여
`ideas/`, `ideas/adopted/`, `ideas/rejected/` 디렉토리의 기존 파일을 모두 확인하여 가장 높은 번호 + 1을 부여한다.

### 작성 범위
`ideas/_template.md` 기반으로 아래 섹션을 작성한다. 평가 점수·판정 섹션은 비워둔다.

| 섹션 | 작성 깊이 |
|------|----------|
| 한줄 요약 | 핵심 가치를 한 문장으로 |
| 해결하려는 문제 | 검색에서 수집한 실제 불만·수요를 근거로 기술 |
| 타겟 사용자 | 주요 타겟, 연령대, 규모 추정 |
| 핵심 기능 | MVP 범위 최대 3개 |
| 수익 모델 | 초기 구상 수준 (2단계에서 검증) |
| 경쟁 현황 | brave-search에서 확인한 유사 앱을 간략히 기록 (2단계에서 심화) |
| 기술 구현 검토 | Expo+RN+Supabase 적합성 및 Windows 개발 환경 호환성 간략 판단 (2단계에서 상세 평가) |

### 검색 근거 인용
수집한 검색 결과를 초안에 반영한다:
- **해결하려는 문제**: naver-search에서 발견한 실제 불만·수요 게시글을 근거로 인용 (fetch로 확인한 본문 포함)
- **경쟁 현황**: brave-search에서 확인한 유사 앱·서비스명 기록
- 이 근거는 2단계에서 재수집 없이 활용된다

|------|------|----------|

## 전환
- 초안 완료, 제외 기준 미해당 → 사용자에게 "2단계: `/stage-2 NNN-아이디어명`" 안내
- 제외 기준 해당 → 폐기 후 새 아이디어 발굴
- 사용자가 종료 요청 → 중단
