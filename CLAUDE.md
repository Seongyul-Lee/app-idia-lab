# App Idea Lab

## 목적
한국 시장 타겟 1인 개발 모바일 앱 아이디어 발굴 및 평가

## 프로젝트 역할
이 프로젝트는 **코드 개발을 하지 않는다.** 아래 4단계를 수행한다.

1. **아이디어 발굴** — 한국 시장 pain point 기반 앱 아이디어 도출
2. **사업성/기술/경쟁력 평가** — 시장 규모, 수익화 가능성, 기술 실현성, 경쟁 환경 분석
3. **PRD 문서 작성** — 채택 가능성이 있는 아이디어에 대해 제품 요구사항 문서 작성
4. **PRD 정합성 판별** — 작성된 PRD의 다각도 검증 (아래 상세 기준 참조)

## 문서 작성 언어
- 모든 문서는 한국어로 작성

## 기술 스택 제약
- React Native + TypeScript
- Supabase (PostgreSQL)
- 외부 파트너십 없이 혼자 구현 가능해야 함

## 평가 우선순위
1. 1-3개월 내 MVP 출시 가능 여부
2. 한국 로컬 시장의 실제 pain point
3. 기존 스택으로 구현 가능한지

## 제외 기준
아래에 해당하는 아이디어는 조사 대상에서 제외한다.
- 법적 인허가가 필수이거나 규제가 까다로운 영역 (의료, 금융 등)
- 기업/기관과의 협약이 필수인 아이디어
- 1인 개발로 구현이 불가능한 아이디어 (예: 다중 쇼핑몰 매출/재고 관리 서비스 — 수십~수백 개 API 연동 및 기업 협약 필요)

## PRD 정합성 판별 기준
PRD 문서 작성 후, 아래 관점에서 다각도 검증을 수행한다.

### 사업성 검증
- **시장 경쟁력**: 기존 경쟁 서비스 대비 진입 가능성
- **차별점 강도**: 경쟁자 대비 명확하고 방어 가능한 차별화 요소 존재 여부
- **수익화 현실성**: 수익 모델의 구체성, 사용자 지불 의향, 단가 적정성
- **TAM/SAM/SOM**: 시장 규모 추정치의 논리적 근거와 현실성

### 리스크 검증
- **핵심 리스크 분석**: 사업/기술/법률/운영 리스크 식별 및 대응 방안
- **의존성 리스크**: 외부 API, 플랫폼 정책 변경 등 통제 불가 요소

### 기술 검증
- **기술 스택 정합성**: React Native + TypeScript + Supabase로 PRD 명세 구현 가능 여부
- **아키텍처 정합성**: 1인 개발 규모에 맞는 설계인지, 과도한 복잡성은 없는지
- **MVP 범위 적정성**: 3개월 내 출시 가능한 범위로 한정되어 있는지

### 문서 품질 검증
- **내적 일관성**: PRD 내 목표, 기능, 기술 명세 간 모순 없음
- **완결성**: 누락된 핵심 섹션이 없음
- **측정 가능성**: 성공 지표(KPI)가 구체적이고 측정 가능한지

> 위 기준은 예시이며, 아이디어 특성에 따라 추가 관점에서 판별할 수 있다.

## 판정 결과 처리

### 통과 (채택)
- `ideas/adopted/`에 저장
- PRD 문서 + **통과 사유 문서** 함께 보관

### 미통과 (탈락)
- **반드시 사용자의 명시적 명령 또는 동의**를 받은 후에만 탈락 처리
- `ideas/rejected/`에 저장
- 관련 문서 + **실패 사유 문서** 함께 보관
- 실패 사유에는 어떤 검증 단계에서 미달했는지 구체적으로 기록

## 프로젝트 구조
```
app-idia-lab/
├── CLAUDE.md
├── ideas/
│   ├── _template.md        # 아이디어 작성 템플릿
│   ├── adopted/            # 채택된 아이디어 (PRD + 통과 사유)
│   └── rejected/           # 탈락된 아이디어 (관련 문서 + 실패 사유)
├── research/
│   ├── competitors/        # 경쟁사 분석
│   └── market-data/        # 시장 데이터
└── evaluation/
    └── criteria.md         # 평가 기준 정의
```

## 아이디어 문서 규칙
- 템플릿: `ideas/_template.md`를 복사하여 작성
- 파일명: `NNN-아이디어명.md` (예: `001-동네-소음-리포트.md`)
- 채택 시 `ideas/adopted/`에, 탈락 시 `ideas/rejected/`에 저장
- 탈락 아이디어도 반드시 판정 사유를 기록

## 워크플로우
아이디어 발굴부터 최종 판정까지 5단계 파이프라인으로 운영한다. 각 단계는 `/stage-N` 커맨드로 실행하며, 단계 간 산출물은 파일로 핸드오프한다.

```
┌─────────┐    ┌─────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1. 발굴  │───→│ 2. 평가  │───→│ 3. PRD   │───→│ 4. 판별  │───→│ 5. 판정  │
└─────────┘    └────┬────┘    │   작성    │    └────┬────┘    │   처리    │
     ↑               │         └─────┬────┘         │         └──────────┘
     │          보류/피벗              │          수정 필요
     └───────────────┘               │              │
                                     └──────────────┘
```

| 단계 | 커맨드 | 핵심 목적 |
|------|--------|----------|
| 1단계: 발굴 | `/stage-1` | Pain point 기반 아이디어 도출 |
| 2단계: 평가 | `/stage-2 NNN-아이디어명` | 6개 항목 정량 평가 (30점 만점) |
| 3단계: PRD 작성 | `/stage-3 NNN-아이디어명` | 제품 요구사항 문서 작성 |
| 4단계: 판별 | `/stage-4 NNN-아이디어명` | PRD 다각도 정합성 검증 |
| 5단계: 판정 처리 | `/stage-5 NNN-아이디어명 채택/탈락` | 채택/탈락/보류 결과 처리 |

모든 단계 전환 시 사용자에게 현재 상태를 보고하고, 다음 단계 진행 여부를 확인받는다.
